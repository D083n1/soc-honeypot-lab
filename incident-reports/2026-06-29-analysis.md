## 2. Detection Engineering

### Custom Rule 100503 — Successful Cowrie Authentication
Confirmed firing correctly in this sample (7/7 successful-login events correctly triggered level-10 alerts with MITRE T1078 tagging).

```xml
<rule id="100503" level="10">
  <if_sid>100500</if_sid>
  <field name="eventid">cowrie.login.success</field>
  <description>Cowrie: INTRUSIONE! Login riuscito da $(src_ip)</description>
  <mitre>
    <id>T1078</id>
  </mitre>
  <group>authentication_success,</group>
</rule>
```

### Global Detection Gaps Identified

**DG-01 — No visibility into failed login attempts.**
The sample contains zero `cowrie.login.failed` alerts, despite seven successful logins using varied, non-trivial credential pairs (`admin/admin`, `root/---fuck_you----`, `zabbix/zabbix`, `root/P4n7z`, `root/admin@8888`, `root/<empty>`). This is consistent with **either** (a) no decoder/rule currently maps `cowrie.login.failed` into a Wazuh alert, making unsuccessful brute-force attempts invisible to the SIEM even though Cowrie logs them locally, **or** (b) the Cowrie `userdb` is configured to accept most/all credentials, so genuine failures rarely occur. Given that several of the observed successful logins used distinct, deliberately-chosen passwords (`P4n7z`, `admin@8888`) typical of dictionary attacks that normally fail many times before succeeding, option (a) is the more likely explanation and should be verified against the decoder configuration.
**Recommendation:** add a rule mapping `cowrie.login.failed` (e.g., level 5–7, grouped for volumetric correlation) so brute-force volume and credential-list reconstruction become visible.

**DG-02 — No dedicated rule for post-authentication SSH tunneling/port-forwarding abuse.**
The `cowrie.direct-tcpip.request` / `cowrie.direct-tcpip.data` events observed in IR-02 (below) are only captured by the generic, low-severity rule `100500` ("Cowrie: Evento di sistema", level 3). A successful login immediately followed by an outbound tunnel request is a materially different — and higher-severity — event than routine telemetry.
**Recommendation:** add a rule (e.g., `100505`, level 8–9) that fires when `cowrie.direct-tcpip.request` follows a `cowrie.login.success` within the same session, mapped to MITRE T1572 (Protocol Tunneling).

**DG-03 — Timestamp discrepancy between Wazuh alert time and Cowrie sensor time.**
Across the sample, the alert-level `timestamp` field (assigned by the Wazuh manager) consistently lags the embedded `data.timestamp` field (assigned by the Cowrie sensor) by approximately **5 to 5.5 hours**, and this gap is not constant — it narrows and widens across the sample rather than holding a fixed offset. This pattern is more consistent with clock drift or an intermittent ingestion delay than with a simple timezone misconfiguration, though a timezone bug cannot be ruled out from alert data alone.
**Recommendation:** verify NTP synchronization on both the honeypot host and the Wazuh manager VM; a fixed offset would point to timezone config, a variable one to clock drift or queuing delay. This matters because incident timelines in this document are reconstructed from the Cowrie sensor's own embedded timestamps, not the Wazuh alert timestamps, specifically to avoid this discrepancy.

**DG-04 — Host-level `sshd` activity is not correlated with honeypot telemetry.**
Four source IPs (`45.79.207.71`, `45.79.115.59`, `172.104.241.98`, `45.33.14.5`) triggered default OSSEC `sshd` rules (`5710`, `5740`, `5762`) against the real operating-system SSH daemon on the same host — separate from the Cowrie honeypot on port 2222. These are currently isolated, low-severity, uncorrelated events with no aggregation rule tying repeated short-lived connection resets from multiple source IPs together as a scanning pattern.
**Recommendation:** add a correlation rule aggregating multiple `sshd` connection-reset/invalid-user events across distinct source IPs within a short window, and consider whether the real `sshd` port should be further restricted given it is already receiving unsolicited probes.

## 3. Incident Reports

### IR-01 — Reconnaissance and Banner Grabbing Without Authentication

**Executive Summary**
Three distinct source IPs opened TCP/SSH connections to the Cowrie honeypot and disconnected without any authentication attempt, consistent with automated internet-wide scanning/banner-grabbing rather than targeted intrusion attempts.

**Timeline** (Cowrie sensor time, UTC)
| Time | Source IP | Session | Behavior |
|---|---|---|---|
| 08:02:35 | 124.174.10.254 | `abb0cd7825cd` | Connect → client version captured → idle ~120s → closed, no login |
| 11:12:36 – 11:14:49 | 211.155.100.9 | 3 sessions | Two closed in ~6s each; one idle ~120s then closed. No login on any session |
| 12:26:17 – 12:26:27 | 218.92.212.222 | 3 sessions | Closed in <300ms, <25ms, and 3.2s respectively. Client version/kex captured on one session only. No login |

**Evidence**
- No `cowrie.login.success` or `cowrie.login.failed` events for any of these three IPs.
- Session durations are either very short (tens of milliseconds to a few seconds) or held open with no activity for exactly ~120 seconds (a value consistent with a client-side connection timeout rather than manual interaction).
- Client fingerprints captured: `SSH-2.0-Go` (211.155.100.9), `SSH-2.0-libssh_0.7.4` (218.92.212.222, hassh `e37f354a101aff5871ba233aa82b84ec`).

**Technical Analysis**
The pattern — connect, optionally negotiate key exchange, disconnect, with no credentials submitted — is consistent with automated SSH banner-grabbing or asset-mapping tools used during internet-wide reconnaissance sweeps. The repeated short-lived connections from `218.92.212.222` within a ten-second window suggest a scripted retry loop rather than a single manual probe.

**MITRE ATT&CK Mapping**
- T1595 — Active Scanning

**Detection Gaps Identified**
See DG-04 (no cross-IP correlation for these short-lived scanning sessions).

**Assessment**
This activity is consistent with routine, low-intent internet background noise rather than a targeted attack against this host specifically. No compromise occurred.

**Confidence Level: High** — the absence of any login event for these sessions is directly and unambiguously supported by the logs; the "automated scanning" interpretation is the most parsimonious explanation for connect-then-disconnect-with-no-auth behavior at this frequency, though it cannot be proven the traffic wasn't manual.

### IR-02 — Successful Authentication Followed by SSH Tunneling Attempt Toward Cloudflare Infrastructure (45.148.10.121)

**Executive Summary**
The source IP `45.148.10.121` authenticated successfully twice using `admin/admin`, and in both sessions immediately issued an SSH `direct-tcpip` (port-forwarding) request toward a Cloudflare-associated IP on port 3478, sending a single HTTP `GET` request, before disconnecting within ~200ms. No interactive shell commands were executed in either session.

**Timeline** (Cowrie sensor time, UTC)
| Time | Session | Event |
|---|---|---|
| 05:47:53 | `7e134830a240` | Connect → closed after 22ms, no login (precursor probe) |
| 06:28:21 | `d6d89edacfbe` | Connect → `client.version: SSH-2.0-Go` → kex → **login success `admin/admin`** → `direct-tcpip.request` to `141.101.90.1:3478` → `direct-tcpip.data`: HTTP GET, `Host: turn.cloudflare.com:3478` → closed after 172ms |
| 06:28:21 | `d46180450adc` | Connect → same client fingerprint → **login success `admin/admin`** → identical `direct-tcpip.request`/`data` sequence to `141.101.90.1:3478` → closed after 174ms |

**Evidence**
- `cowrie.login.success`: `username=admin`, `password=admin`, fired twice (rule `100503`, level 10).
- `cowrie.direct-tcpip.request`: destination `141.101.90.1:3478`, fired in both post-login sessions.
- `cowrie.direct-tcpip.data`: payload `GET / HTTP/1.1\r\nHost: turn.cloudflare.com:3478\r\nConnection: close\r\n\r\n`, explicitly logged as **discarded** by Cowrie (the honeypot did not actually relay the connection).
- Client fingerprint consistent across both sessions: `SSH-2.0-Go`, hassh `bf7dbf67fa9b26ee9f9deb99c49320ba`.
- No `cowrie.command.input` events in either session — no interactive shell activity.

**Technical Analysis**
`141.101.90.1:3478` resolves in the captured HTTP `Host` header to `turn.cloudflare.com`, a Cloudflare TURN/relay endpoint typically used for WebRTC traffic relaying. Requesting an SSH `direct-tcpip` forward to this destination immediately after login, without any preceding interactive reconnaissance, suggests an automated tool testing whether the compromised SSH session can be used to relay/tunnel outbound traffic — a pattern associated with abusing compromised SSH hosts as open relays or proxies rather than with manual, hands-on-keyboard exploitation. The identical behavior across two back-to-back sessions from the same client fingerprint supports an automated, scripted actor.

**MITRE ATT&CK Mapping**
- T1110.001 — Password Guessing (credential used: `admin/admin`)
- T1078 — Valid Accounts (post-compromise reuse of guessed credentials)
- T1572 — Protocol Tunneling (SSH port-forwarding used to reach an external endpoint)
- *Possible, less certain:* T1090 — Proxy (if the intent was to use the host as a relay; not directly provable from a single discarded HTTP GET)

**Detection Gaps Identified**
See DG-02 — this sequence was only logged at generic informational severity (rule `100500`, level 3) despite representing tunneling activity on an already-compromised session.

**Assessment**
This behavior is consistent with automated, credential-guessing tooling that verifies tunneling/relay capability on newly compromised hosts as a standard post-exploitation check, rather than manual attacker activity. No evidence of data exfiltration or successful relay was observed — Cowrie explicitly discarded the forward request.

**Confidence Level: High** — the login, destination, and payload are all directly present and consistent across two independent sessions from the same fingerprint. The specific intent behind the tunneling attempt (proxy abuse vs. generic connectivity test) is inferred and therefore held to Medium confidence.

### IR-03 — Successful Authentication with System Fingerprinting Command (120.48.57.228)

**Executive Summary**
The source IP `120.48.57.228` authenticated successfully with a profanity-laced password and executed a single reconnaissance command (`uname -s -m`) before disconnecting, without deploying any further payload in this session.

**Timeline** (Cowrie sensor time, UTC)
| Time | Session | Event |
|---|---|---|
| 09:48:35 | `eb5c02a75406` | Connect → closed after 320ms, no login (precursor probe) |
| 09:48:36 | `fd2d4d6c6e4a` | Connect → `client.version: SSH-2.0-Go` → kex → **login success `root/---fuck_you----`** → `session.params` (arch `linux-x64-lsb`) → **command: `uname -s -m`** → TTY log closed (13 bytes) → session closed after 6622ms |

**Evidence**
- `cowrie.login.success`: `username=root`, `password=---fuck_you----` (rule `100503`, level 10).
- `cowrie.command.input`: `uname -s -m` (rule `100502`, level 7, tagged T1059).
- Reported client architecture: `linux-x64-lsb`.
- No further commands, downloads, or file-write activity captured in this session.

**Technical Analysis**
`uname -s -m` reports the operating system name and CPU architecture. This is a low-cost, high-value reconnaissance step commonly performed by automated exploitation frameworks to select an architecture-appropriate payload before dropping a second-stage binary (e.g., choosing an ARM vs. x86_64 build of a botnet or cryptomining agent). No such follow-up download or execution was observed in this specific session — either the session was cut short before the actor's script continued, or the tool only performs fingerprinting during this stage and stages payload delivery separately.

**MITRE ATT&CK Mapping**
- T1110.001 — Password Guessing
- T1078 — Valid Accounts
- T1082 — System Information Discovery
- T1059 — Command and Scripting Interpreter

**Detection Gaps Identified**
None specific to this incident beyond DG-01 (no visibility into any failed attempts that may have preceded this successful login).

**Assessment**
Consistent with automated, scripted exploitation tooling performing standard pre-payload reconnaissance. The short session duration and single-command pattern suggest a lightweight probe rather than sustained hands-on-keyboard activity, though this cannot be confirmed without observing a follow-up session from the same actor.

**Confidence Level: Medium-High** — the command and login are directly evidenced; the inference that this fingerprinting precedes payload staging is a reasonable, commonly-observed pattern but is not directly proven within this single session.

### IR-04 — Repeated Credential Guessing with Automated Post-Login Verification Command (185.242.3.195)

**Executive Summary**
The source IP `185.242.3.195` successfully authenticated twice, roughly 43 minutes apart, using two different credential pairs (`zabbix/zabbix` and `root/admin@8888`), and in both cases executed the identical harmless command `echo OK` before disconnecting within under half a second.

**Timeline** (Cowrie sensor time, UTC)
| Time | Session | Event |
|---|---|---|
| 10:33:54 | `2e0971cb22e1` | Connect → **login success `zabbix/zabbix`** → **command: `echo OK`** → closed after 293ms |
| 12:06:01 | `8fce1e5af033` | Connect → **login success `root/admin@8888`** → **command: `echo OK`** → closed after 347ms |

**Evidence**
- Two `cowrie.login.success` events (rule `100503`) with distinct usernames/passwords from the same source IP.
- Two `cowrie.command.input` events, both with identical payload `echo OK` (rule `100502`).
- Client fingerprint (`SSH-2.0-Go`, hassh `16443846184eafde36765c9bab2f4397`) identical across both sessions, confirming both originate from the same tool/host.
- Session durations under 400ms in both cases — no further interaction.

**Technical Analysis**
Executing a fixed, harmless command immediately after login and then disconnecting is a well-known automated-tooling signature used to verify that a guessed credential pair grants working shell access, without yet committing to further exploitation in the same session. The credential `zabbix/zabbix` targets the default account of the Zabbix monitoring platform, suggesting the tool may be part of a broader campaign cycling through service-specific default credentials rather than SSH-specific ones only.

**MITRE ATT&CK Mapping**
- T1110.001 — Password Guessing
- T1078 — Valid Accounts
- T1059 — Command and Scripting Interpreter (minimal, verification-only)

**Detection Gaps Identified**
None specific beyond DG-01.

**Assessment**
Consistent with a credential-validation/access-brokering tool that confirms working logins for later reuse (e.g., resale of access, or hand-off to a separate exploitation stage) rather than an actor performing exploitation directly in this session.

**Confidence Level: High** for the credential-guessing and verification-command pattern (directly evidenced and repeated); **Medium** for the "access brokering" interpretation of intent, which is inferred from the behavior pattern rather than stated anywhere in the logs.

### IR-05 — Suspected Internal/Analyst Testing Activity, Not Classified as External Threat (Source IP Redacted)

**Executive Summary**
A single source IP — **redacted from this document as a matter of standard OPSEC practice, since it was assessed as belonging to the analyst's own testing infrastructure rather than an external party** — accounts for the largest share of Cowrie alerts in this sample (26 of 90) and displays technical characteristics markedly different from every other source IP observed: a genuine interactive OpenSSH client, a negotiated terminal size, an Italian system locale, and human-paced command entry. **The analyst has indicated this traffic is likely their own prior testing of the honeypot, not recalled with certainty.** This incident is documented for completeness but should not be treated as an external threat.

**Timeline** (Cowrie sensor time, UTC)
| Time | Session | Event |
|---|---|---|
| 10:44:30 | `5122d73c403b` | Connect → `client.version: SSH-2.0-OpenSSH_10.2` → kex → **login success `root/P4n7z`** → terminal size negotiated (80×24) → `LANG=it_IT.UTF-8` → several empty command-input events → **command: `exit`** → session closed after 25.3s |
| 10:45:52 | `7b4662134324` | Connect → same client fingerprint → **login success `root/<empty password>`** → terminal size (80×24) → `LANG=it_IT.UTF-8` → empty command-input events → **command: `ls -la`** → more empty command-input events → session closed after 11.3s |

**Evidence**
- Client version `SSH-2.0-OpenSSH_10.2` — a standard, real interactive OpenSSH client, in contrast to every other source IP in this sample, all of which reported automated client libraries (`SSH-2.0-Go`, `SSH-2.0-libssh`).
- `cowrie.client.size`: terminal explicitly negotiated at 80×24 — a step only performed by interactive terminal sessions, not scripted tools.
- `cowrie.client.var`: `LANG=it_IT.UTF-8` — an Italian-language locale environment variable.
- Multiple empty `cowrie.command.input` events interspersed with the two real commands (`exit`, `ls -la`) — consistent with a human pressing Enter, retyping, or navigating interactively rather than a scripted, single-shot payload.
- Two different credential pairs used within a two-minute window from the same source IP.

**Technical Analysis**
Every other successful-login source IP in this sample used an automated client library and completed its entire session in under seven seconds with zero or one command. The redacted IP, by contrast, used a real OpenSSH client, negotiated a terminal, ran ordinary exploratory shell commands (`ls -la`), and kept sessions open for 11–25 seconds — a profile much closer to a human manually testing the honeypot than to opportunistic scanning malware. The Italian system locale is also notable given the honeypot operator's own locale.

**MITRE ATT&CK Mapping**
Not applicable for this incident. Given the strong indicators of legitimate testing, mapping this activity to adversary tactics would misrepresent it; it is documented here as a data-provenance note rather than a threat.

**Detection Gaps Identified**
There is currently no mechanism (allowlist, tagging, or dedicated rule) to distinguish known analyst-testing source IPs from genuine external traffic, which risks inflating apparent "successful compromise" counts and skewing statistics (as seen in Section 1, where this single IP accounts for ~29% of all Cowrie alerts in the sample).
**Recommendation:** maintain an allowlist of known testing source IPs/ranges and exclude or clearly tag their alerts in reporting and dashboards.

**Assessment**
This activity is far more consistent with legitimate operator testing of the honeypot than with a genuine external attacker, based on client fingerprint, terminal negotiation, locale, and interaction pacing. It should be excluded from threat statistics unless the analyst can rule out having generated it.

**Confidence Level: Low** (as an external threat) — deliberately downgraded given the strong behavioral indicators of self-testing and the analyst's own uncertainty about the source. If confirmed as analyst activity, this should be relabeled as a testing/validation log entry rather than an incident.

## 4. Cross-Incident Correlation Notes

This does not by itself prove common operatorship.
- **OS-level `sshd` probing is temporally interleaved with honeypot activity** but not currently correlated in tooling (see DG-04); all four host-level `sshd` source IPs (`45.79.207.71`, `45.79.115.59`, `172.104.241.98`, `45.33.14.5`) reset the connection before completing key exchange or attempted a single invalid username, with no successful authentication against the real daemon.
